# Excel WebSocket 双进程展示程序

## 1. 项目简介
这个项目的目标是把 Excel 数据用桌面界面展示出来，并且严格按“两个独立进程 + WebSocket 通信”的方式实现。

核心文件：
- `read_data.py`：负责读取 `data.xlsx`，转换为二维数组并提供 WebSocket 服务
- `show_data.py`：负责连接 WebSocket，接收二维数组并用 PyQt 表格展示
- `app.py`：总控入口，负责按顺序拉起 `read` 和 `show` 两个子进程，并管理退出

## 2. 功能说明
- 读取 Excel 数据为 Python 二维数组（第一行作为表头）
- 读取进程和展示进程独立运行
- 通过 WebSocket 完成进程间通信
- 使用 PyQt UI 展示表格数据
- 程序单实例运行，重复启动会提示“程序已启动”

## 3. 运行方式
### 3.1 推荐方式：总控启动
```powershell
py -3.13 app.py
```
或直接运行打包后的：
```powershell
.\python_research.exe
```

### 3.2 分开启动两个进程
先启动读取进程：
```powershell
py -3.13 read_data.py
```
再启动展示进程：
```powershell
py -3.13 show_data.py
```

## 4. 数据文件要求
- 文件名固定：`data.xlsx`
- 文件需与程序在同一目录
- 默认读取第一个工作表
- 第一行是表头，后续行为数据

## 5. 项目实现思路
这个项目采用了“职责拆分”的设计：

1. 数据读取和界面展示分离  
`read_data.py` 专注读 Excel；`show_data.py` 专注 UI。这样每个文件只做一件事，逻辑更清晰。

2. 用 WebSocket 作为进程通信通道  
读取进程作为服务端，展示进程作为客户端。展示进程拿到消息后直接渲染表格。

3. 使用 `app.py` 做调度器  
主入口先拉起读取进程，再拉起展示进程，保证展示时服务端已经就绪。展示窗口关闭后，主入口会回收读取进程。

4. 单实例控制  
通过绑定固定本地端口实现“进程锁”。同类进程二次启动时端口会被占用，从而触发提示并退出。

5. 兼容源码运行和打包运行  
读取 `data.xlsx` 时同时兼容普通脚本路径和 PyInstaller onefile 的临时解压路径（`sys._MEIPASS`）。

## 6. 开发中遇到的问题与解决方案
### 问题 1：重复启动看起来“没反应”
原因：打包后是无控制台模式，重复启动时程序直接退出，用户看不到日志。  
解决：在单实例检测失败时加入弹窗提示（主程序显示“主程序已启动！”）。

### 问题 2：Nuitka 在当前环境编译不稳定
现象：出现编译异常，甚至影响系统稳定性。  
解决：改用 PyInstaller 单文件打包方案，稳定性更高、发布更直接。

### 问题 3：打包后找不到 `data.xlsx`
原因：onefile 运行时资源路径与源码路径不同。  
解决：增加统一资源目录函数，优先判断 `sys._MEIPASS`，否则回退到脚本目录。

### 问题 4：展示进程偶发连不上读取进程
原因：展示进程启动太快，读取服务端还未完成监听。  
解决：展示端增加短时重试机制，主入口中也增加“先读后显”的启动顺序和短延时。

## 7. 目录结构（主要文件）
- `app.py`：主入口（launch / read / show 三模式）
- `read_data.py`：读取 Excel + WebSocket 服务端
- `show_data.py`：WebSocket 客户端 + PyQt 表格 UI
- `build_exe.bat`：一键构建脚本
- `data.xlsx`：示例数据文件

## 8. 常见问题
### Q1：这个程序只能读取当前这份 Excel 吗？
不是。你可以替换 `data.xlsx` 的内容，程序会按新表格结构展示。

### Q2：为什么要拆成两个进程，不直接一个文件全做？
因为需求明确要求“两个独立进程 + 进程间通信”，这个结构是按需求设计的。
